Make two lead pages public in Cloudflare Access

This checklist walks you, step by step, through making just two clients' lead pages open to anyone with the link — Dan Blankenship and OC Dents — while every other client's pages keep asking visitors to log in. You do all of this in the Cloudflare Zero Trust dashboard. There is no file to edit, no website to re-publish, and nothing to commit. Your changes go live within seconds of saving.

Time needed: about 10 minutes Where: Cloudflare Zero Trust dashboard (browser) Risk: low — fully reversible (see Rollback)

1 What this does — and what it won't touch

The leads website lives at leads.branchandrootconsulting.com. Right now, a single Cloudflare Access "catch-all" rule sits in front of that whole address and asks every visitor to sign in before they can see anything. You are going to add two small, narrowly-targeted exceptions — one for Dan Blankenship's pages, one for OC Dents' pages — that let those (and only those) pages load with no login. This works because Cloudflare always obeys the most specific rule for a given web address, so a rule aimed at one client's folder quietly overrides the broad catch-all gate for that folder alone.

This leaves the following completely alone:
  • The site-wide login gate — the existing catch-all application that protects everything stays exactly as it is. You will not edit or delete it.
  • The existing public bypass for mini-audits and questionnaires — there is already a rule that makes any */mini-audit* and */questionnaire* page public for every client. Do not touch it. It keeps doing its job.
  • Every other client — all other demos, audits, and proposals stay behind the login screen, just as they are today.

2 Before you start

Get yourself into the right screen and find your bearings. Tick each box as you go.

A quick note on wording Cloudflare occasionally renames buttons. If you don't see the exact label, look for the closest match (for example "Add an application" may read "Create application"). The order of the steps stays the same.

3 Add application 1 — Dan Blankenship

You're creating a brand-new Access application that points only at Dan's pages and waves visitors straight through.

  1. Start a new application. On the Applications screen, click Add an application.
  2. Choose the type. Select Self-hosted.
  3. Name it. In the application name field, type exactly:
    Application name
  4. Set the two application paths. Find the Application domain / paths area. You need to add two entries, both on the host leads.branchandrootconsulting.com. Add the first, then use "Add" (or the +) to add the second. Click each box below to copy the exact path:
    Path 1 (the page itself)
    Path 2 (everything under it)

    You need both. The first covers the landing page on its own (the demo). The second — with the /* on the end — covers every sub-page beneath it, such as the audit and the images in the /media/ folder. If your screen splits the host and the path into two separate boxes, put leads.branchandrootconsulting.com in the host box and /round-1/dan-blankenship (then /round-1/dan-blankenship/*) in the path box.

  5. Move on to the policy. Continue (e.g. Next) until you reach the policies step. Click Add a policy (or use the default policy block).
  6. Set the policy to let everyone in. Give the policy any clear name (for example Public bypass), then set these two values exactly:
    Action Bypass
    Include → Selector Everyone

    "Bypass" means "skip the login for these pages." "Everyone" means "for all visitors." Together they make the pages public.

  7. Save. Finish the wizard and click Save / Add application. You'll land back on the Applications list with Public — Dan Blankenship now showing.
Double-check the Action before saving The Action must be Bypass. If it's left on "Allow", visitors could still be asked to authenticate. Bypass is the setting that makes a page truly public.

4 Add application 2 — OC Dents

Exactly the same process as above, with OC Dents' name and paths. Repeat every step.

  1. Start a new application. Click Add an application.
  2. Choose the type. Select Self-hosted.
  3. Name it. Type exactly:
    Application name
  4. Set the two application paths. Add both entries on the host leads.branchandrootconsulting.com:
    Path 1 (the page itself)
    Path 2 (everything under it)

    Same reasoning as before: the bare path covers the demo landing page; the /* version covers the audit, the proposal, and the /media/ images.

  5. Move on to the policy. Continue to the policies step and click Add a policy.
  6. Set the policy to let everyone in. Name it (e.g. Public bypass) and set:
    Action Bypass
    Include → Selector Everyone
  7. Save. Finish and click Save. You should now see both new applications — "Public — Dan Blankenship" and "Public — OC Dents" — in your list.

6 Rollback — re-gate instantly

If you ever need Dan's or OC Dents' pages to be private again, you simply delete the matching application. The moment it's gone, the site-wide login gate takes back over for those pages — usually within seconds. You are not undoing anything else.

Only delete the two applications you created Never delete the site-wide catch-all application or the existing mini-audit / questionnaire bypass. Removing those would change behaviour for every client. Delete only "Public — Dan Blankenship" and "Public — OC Dents".

7 Troubleshooting

If a page that should be public is still asking visitors to log in, work down this list:

  • Check the path spelling. The path is case- and character-sensitive. It must read exactly /round-1/dan-blankenship or /round-1/oc-dents — with a hyphen, no spaces, no .html, and no trailing slash. A single typo means the rule doesn't match the page.
  • Make sure BOTH paths exist on the application. Open the application and confirm it lists both the bare path and the /* version. If only the bare path is there, the landing page works but the audit, proposal, and images will still be gated (and vice-versa).
  • Remember that the most specific path wins. Cloudflare Access applies the rule whose path most closely matches the page being visited. Your two new applications are more specific than the site-wide catch-all, which is exactly why they override it for those folders. If a page is still gated, it usually means no specific rule matched it — so the broad gate is correctly taking over. Re-check the path.
  • Give it a few seconds. Changes are near-instant but not always instantaneous. Wait 15–30 seconds after saving, then try again.
  • Use a fresh private window or clear your cache. Your browser may be holding an old login session or a cached copy of the page. Open a new private / incognito window (close any old ones first), or clear the browser cache, then re-test. This rules out a stale result rather than a real problem with the rule.
  • Confirm the Action is "Bypass", not "Allow". Open the application's policy and verify the Action reads Bypass with Include set to Everyone. "Allow" can still trigger an authentication step.
When in doubt, roll back If something looks wrong and you can't pin it down, it is always safe to delete the two new applications (section 6). That returns everything to the known-good "everything is gated" state, and you can try again with a clear head.
